# NexaCard OTP Service Design

## Goal

Build an independent local service that receives a card number, card type, and
upstream order creation time, then returns the matching NexaCard payment OTP.
The service must use the local native Google Chrome installation, run headless
by default, connect directly without any proxy or fingerprint browser, and
recover an expired NexaCard session only when an OTP request discovers that the
session is no longer authenticated.

The service queries OTPs only. It does not initiate payments or modify cards,
orders, balances, or transactions.

## Confirmed NexaCard Behavior

The authenticated site exposes separate verification-code pages:

- NexaCardB: `/nova-v-card-b/verify-code`, API `channel_id=8`.
- 3D-1 card: `/3d-1-card/verify-code`, API `channel_id=6`.

Both pages provide exact-card search and return rows containing:

- six-digit `verify_code`;
- full `virtual_card`;
- bare `created` timestamp formatted as `YYYY-MM-DD HH:mm:ss`;
- optional `trade_time`, which is often empty.

The API timestamp contains no timezone suffix. In the inspected Chrome session,
the browser timezone was `Asia/Shanghai`, and the UI displayed the API's
`created` value unchanged. The service therefore interprets NexaCard page times
using an explicit configurable timezone, defaulting to `Asia/Shanghai`, rather
than relying on the host browser's implicit timezone.

NexaCard API calls include dynamic authorization, token, timestamp, and
signature values generated by the site. The service deliberately automates the
official pages instead of reimplementing that private signing scheme.

## Architecture

Create a focused `nexacard_otp` package with these boundaries:

- `config`: validates current WebUI/.env values and fixed private-file paths.
- `browser`: owns the native Chrome process, persistent context, and per-request
  pages.
- `gmail_auth`: starts Google OAuth, handles its callback, persists credentials,
  validates the authorized Gmail address, and reports authorization status.
- `gmail_reader`: refreshes access tokens when needed and finds the fresh
  NexaCard login-verification email.
- `nexacard_login`: detects an unauthenticated NexaCard page and performs the
  email-verification login flow.
- `otp_lookup`: searches the correct card page, applies time matching, and
  performs bounded polling.
- `api`: exposes the local HTTP contract and maps domain failures to HTTP
  responses.

The service is a standalone FastAPI process with configurable bind host and
port. Defaults bind only to `127.0.0.1`. The existing WebUI owns configuration,
Google authorization controls, authorization status, and service start/status
controls, but the OTP API does not depend on a WebUI browser tab remaining open.

## Native Chrome and Direct Networking

Playwright launches the installed Google Chrome executable, not bundled
Chromium and not BitBrowser, AdsPower, Octo, or another fingerprint-browser
provider.

The browser launch contract is:

- headless enabled by default and configurable in WebUI;
- persistent project-private Chrome profile;
- `--no-proxy-server`;
- `--proxy-server=direct://`;
- `--proxy-bypass-list=*`;
- proxy-related environment variables removed from the Chrome child
  environment, including upper- and lower-case HTTP, HTTPS, and ALL proxy
  variables.

This explicit isolation is required because the existing WebUI process may set
proxy environment variables for unrelated project features.

Each OTP request receives its own page in the shared authenticated browser
context so concurrent card searches cannot overwrite one another's form state.
A single login lock protects session recovery. When concurrent requests detect
logout, the first lock holder logs in; later holders re-check authentication
and reuse the recovered session.

## Configuration and WebUI

Add a `NexaCard OTP` WebUI configuration group with:

- `NEXACARD_ACCOUNT`: NexaCard account name.
- `NEXACARD_PASSWORD`: secret NexaCard password field.
- `NEXACARD_VERIFICATION_EMAIL`: Gmail address used by NexaCard login.
- `NEXACARD_HEADLESS`: boolean, default `true`.
- `NEXACARD_CHROME_PATH`: optional explicit Chrome executable path; normal
  Windows Chrome locations are auto-detected when empty.
- `NEXACARD_PAGE_TIMEZONE`: IANA timezone, default `Asia/Shanghai`.
- `NEXACARD_OTP_POLL_INTERVAL_SECONDS`: positive number, default `3`.
- `NEXACARD_OTP_MAX_ATTEMPTS`: positive integer, default `100`.
- `NEXACARD_SERVICE_HOST`: default `127.0.0.1`.
- `NEXACARD_SERVICE_PORT`: local service port.

The polling fields are labeled `OTP 刷新间隔（秒）` and `OTP 最大查询次数`.
The WebUI shows the approximate maximum wait. The first lookup is immediate;
the service waits between subsequent attempts, so the default duration is
approximately five minutes.

Configuration is read at the start of every OTP request. Polling and timezone
changes apply to the next request without restarting the service. Changes to
browser-affecting settings or NexaCard credentials invalidate the current
browser generation; the next request recreates the browser context safely.

## Private Google OAuth Files

Google OAuth files use fixed project-local paths and are not user-editable
WebUI fields:

- `nexacard_otp/private/credentials.json`;
- `nexacard_otp/private/token.json`;
- `nexacard_otp/private/token.meta.json`.

The whole private directory, its Chrome profile, and temporary OAuth state are
excluded from Git. The implementation may migrate the existing local files
from `D:\Gmail API李\credentials.json` and `D:\Gmail API李\token.json` when the
project-private copies do not yet exist. Migration never prints token contents
and never commits either file.

The existing reference reader confirms that successful Google consent writes
`token.json`; when an access token expires, it refreshes with the stored refresh
token and rewrites the file. The new service preserves that lazy refresh
behavior.

## Google Authorization UI and Lifecycle

The verification-email row includes:

- `Google 鉴权`;
- `重新鉴权`;
- `检测状态`;
- authorization email, state, and estimated refresh-token expiration.

Starting authorization opens Google's consent page directly with:

- Gmail read-only scope;
- `login_hint` set to the configured verification email;
- offline access;
- consent prompting, so reauthorization can issue a fresh refresh token;
- a cryptographically random `state` value validated by the local callback.

After the callback exchanges the authorization code, the backend calls Gmail
Profile and verifies that the authorized address exactly matches the configured
verification email. A mismatch is rejected before replacing the active token.
Successful credentials are written atomically to `token.json`.

`token.meta.json` contains no access or refresh token. It records the authorized
email, authorization time, optional `refresh_token_expires_in`, and calculated
refresh-token expiration. If Google does not return the remaining refresh-token
lifetime, the WebUI labels a seven-day Testing-mode value as `预计到期`, not as
an authoritative deadline.

The access-token `expiry` in `token.json` is not presented as the Gmail grant's
expiration. Access tokens are refreshed lazily. Refresh-token validity is
classified as follows:

- missing, malformed, or lacking a refresh token: `未鉴权`;
- valid access token or successful refresh: `授权可用，Access Token 自动续期`;
- refresh rejected with `invalid_grant`, or a confirmed invalid-credentials
  response after one refresh attempt: `授权已失效，请重新鉴权`;
- network, DNS, or temporary Google failure: `暂时无法验证授权状态`.

Status is checked once when the WebUI authorization section loads, when the
operator clicks `检测状态`, or when Gmail access is actually required. There is
no background authorization polling.

## NexaCard Login Recovery

The service never polls NexaCard merely to check login state. During an OTP
request, it first attempts the requested verification-code operation. A login
page, login-form marker, or confirmed authentication failure starts recovery.

Recovery performs:

1. Acquire the shared login lock.
2. Re-check the session because another request may already have recovered it.
3. Fill the configured account and password.
4. Select email verification and enter the configured Gmail address.
5. Record the send-request time, then click `获取验证码`.
6. Query Gmail for NexaCard verification mail received after that time.
7. Require the expected sender/subject and extract the nine-digit login code
   from the message body.
8. Submit the code and verify that an authenticated route is reached.
9. Resume the original OTP request.

The observed sample message uses subject `NexaCard Verification Code`, sender
`jushihui@mail.jushipay.com`, and an HTML body containing a prominent
nine-digit code. The extractor must validate all three characteristics and
must not reuse older unread messages.

Login recovery does not consume an OTP lookup attempt. A single recovery retry
is allowed for one OTP request; repeated authentication failure returns a
bounded error instead of looping.

## OTP API Contract

Expose:

```http
POST /v1/otp
Content-Type: application/json
```

Request:

```json
{
  "card_number": "full card number",
  "card_type": "NexaCardB",
  "order_created_at": "2026-07-19T05:30:20+08:00"
}
```

Accepted card types normalize case, punctuation, and common Chinese aliases to
the two canonical values `NexaCardB` and `3D-1卡`. Card numbers normalize spaces
and hyphens, then must contain only a valid card-length sequence of digits.

Timezone-aware ISO 8601 order timestamps are preferred. A timestamp without a
zone is interpreted in `NEXACARD_PAGE_TIMEZONE`. NexaCard's bare `created`
values are interpreted in that same timezone.

Successful response contains only the requested value:

```json
{"otp":"123456"}
```

No card, timestamps, attempt count, or session metadata is returned on success.

## OTP Matching and Polling

For each attempt, the service:

1. Navigates to the card type's verification-code page.
2. Detects and recovers logout if necessary.
3. Searches by the normalized full card number.
4. Reads all relevant result pages rather than assuming the first ten rows are
   sufficient.
5. Keeps only rows whose full card number exactly matches and whose `created`
   time is strictly later than `order_created_at`.
6. Sorts candidates by the positive time difference and returns the smallest.
   A deterministic ID order breaks an exact timestamp tie.

If no candidate exists, the service waits
`NEXACARD_OTP_POLL_INTERVAL_SECONDS` and repeats until the configured total
attempt count is reached. Attempt one runs immediately. Logout recovery and
bounded transient network retries do not consume an OTP attempt. The polling
loop re-reads current configuration only at request start so one in-flight
request has stable bounds.

## Error Contract

- `400`: invalid card number, card type, or order timestamp.
- `502`: NexaCard login failed or the site returned an unusable page.
- `503`: Gmail authorization is absent/invalid, or a required dependency is
  temporarily unavailable.
- `504`: no matching OTP after the configured maximum attempts.

Temporary network errors receive bounded retries inside the current request.
They are not classified as revoked Gmail authorization. Error responses contain
safe error codes/messages and never include secrets.

## Security and Logging

- Bind locally by default.
- Do not log NexaCard passwords, full card numbers, Gmail login codes, payment
  OTPs, OAuth tokens, request authorization headers, or API signatures.
- Redact card numbers to a small prefix/suffix in operational logs.
- Keep Google credentials, tokens, OAuth metadata, Chrome profiles, screenshots,
  and runtime artifacts out of Git.
- Write tokens atomically so interruption cannot leave a partially written
  credential file.
- Use the minimum Gmail read-only scope.
- Do not expose browser debugging ports beyond loopback.

## Testing and Acceptance

Unit tests cover:

1. Card-type aliases and route mapping.
2. Card-number normalization and exact matching.
3. Strictly-after timestamp behavior and timezone conversion.
4. Nearest-positive-time selection and deterministic ties.
5. Polling boundaries, including exactly the configured maximum attempts.
6. Nine-digit NexaCard login-code extraction from the supplied EML fixture.
7. Gmail access-token refresh and refresh-token failure classification.
8. Configuration validation and immediate next-request application.

Browser and service tests cover:

1. Native Chrome headless launch with direct networking and no proxy inheritance.
2. OTP retrieval from an already authenticated session.
3. On-demand NexaCard re-login followed by transparent request resumption.
4. Gmail authorization failure without an infinite retry.
5. NexaCardB and 3D-1 page selection.
6. Concurrent pages without cross-request search-state leakage.
7. Success responses containing only `otp`.
8. Safe timeout and error responses after configured bounds.

WebUI acceptance covers:

1. Saving account, password, verification email, headless, timezone, and polling
   settings.
2. Google authorization, reauthorization, and status controls.
3. Authorized-email and estimated-expiration display.
4. New `x`/`y` values taking effect on the next request without service restart.

Live acceptance first validates matching against existing read-only records,
then exercises visible-mode login recovery with operator consent, and finally
confirms the same flow in headless mode. No acceptance step initiates a payment.
